Exercise 1 - Using Directory Services to Administer User and Group Accounts

Active Directory is a central repository that contains various objects, such as users, groups, and computers. Active Directory maintains the central control of the users, groups, computers, and services running within a domain. You can configure sites, domains, forests, and organizational units (OU) to manage the Active Directory. For example, the root is always the forest, which can have one or more domains. A domain has distinct security boundaries. You can have sites, which can contain one or more domains, or a domain can contain multiple sites. If your organization has multiple departments, then you can create separate OUs for each department.

In Active Directory, one of the key tasks is managing users and groups. There are two types of users in Windows. The first type is the domain user, who can log on to any domain system in the domain with restrictive permissions. The domain users are centrally managed in a system known as the domain controller. The domain users can be managed using a graphical interface known as Active Directory Users and Computers. You can also manage the domain users using Windows PowerShell.

The other type of user is a local user, who is restricted to the system in which it was created. The local user cannot perform a task outside the system in which it was created. It is also important to note that a local user cannot be part of any domain group, such as Domain Users. Its scope is strictly limited to the local computer on which it is created.

In this exercise, you will learn to create local and domain users.

Exercise 2 - Authentication Methods

Authentication is the process of validating a user’s identity. When a user attempts to log on to a system, user credentials are validated to verify the user’s identity. Before the user is granted access to any object on the network or even on the system, the user’s identity must be validated using the authentication process. For example, a user must know not only the username but also the password. The combination of both are validated in the authentication process. However, the authentication process can also include other factors, such as a second factor on which the user is authenticated. The user must also provide a one-time password (OTP) that is sent to the user. Once the user enters the username, password, and OTP, the user is authenticated.

There are different types of authentication methods. Some of the key methods are:

  • Password-based
  • Multi-factor
  • Certificate-based
  • Biometric

The password-based method is the most widely used method. In this exercise, you will learn about various authentication methods.

Federation

Often there is a need for an organization to conduct business with another organization or even a vendor. The organization may need to establish network connectivity with another organization and share resources on the network. To be able to do this, you need to establish a relationship with the other organization’s network and create security domains to be able to share network resources. The security domains that help you share resources with another organization is called the federation process.

In federation, two domains are required. The first domain is the Resource Partner, and the second domain is the Accounts Partner. When an organization is sharing network resources with the other organization, it is called the Resource Partner. For example, a Web application or certain network shared folders may be granted access to the other organization, which is the Account Partner. The users who will be accessing the resources are located in the Account Partner domain.

Attestation

A hardware root of trust serves as the foundation to secure operations of a system. It serves two key functions:

  • store encryption keys that are used in the cryptographic functions
  • enable a secure boot process

Root of trust is implemented in the hardware and, therefore, cannot be tampered with. In a normal scenario, a system is prone to malware and other types of attacks, and if the encryption keys are stored in the system, they can be easily stolen. However, the root of trust is immune to a malware attack.

Root of trust can be implemented in two different ways. It can be either:

  • a stand-alone security module
  • security module that can be embedded within a processor or system on chip (SoC)

There are two types of hardware root of trust:

  • Fixed function: designed to perform a specific function, such as data encryption. Other functions it can perform is key management and certificate validation. However, other than the defined functions, it cannot perform any other functions.
  • Programmable: is designed to perform complex tasks and functions. It is possible to upgrade a programmable root of trust.

A root of trust can be applied using:

  • A private cryptographic key
  • Physically Unclonable Function (PUF)

With the root of trust enforced, the booting process of a system is considered to be genuine, with any applications running on it.

Motherboards of the modern systems include a chip that is known as a Trusted Platform Module (TPM), which provides a hardware root of trust for secure computing. When a system has TPM, you can configure it by accessing BIOS within the system. A TPM can perform various functions, such as:

  • Storage of encryption keys
  • Management of encryption keys
  • Perform encryption and decryption process
  • Storage of certificates
  • Storage of passwords

Using the TPM, SecureBoot ensures only the trusted operating system bootloaders can load in the system. For the Secureboot to work, you need to have two components in the system:

  • UEFI firmware
  • TPM

Just like the secure boot, Measured Boot Chain also works with the Root of Trust. In the Measured Boot process, each object in the hierarchy is validated. It starts with the Root of Trust, and then the next object in the chain is computing its hash. In this process, the current object calculates the hash of the next object and stores safely. Using this method, a chain of objects is calculated, and their hashes are computed. The hashes can be reported to a management station, known as Attestation. When referring to the Measured Boot, the hashes are stored in the TPM, the Trusted Platform Module.

When a system boots, each object’s hash is compared with the hashes stored in TPM. If every hash meets, the system boots up.

Authentication, Authorization, and Accounting (AAA)

When a user needs to access certain resources on a network, the user needs to be authenticated and then authorized. After the authorization, the user’s activities are logged and tracked using the accounting method.

When a user attempts to connect to a network, the user needs to be identified, which is the identification process. The user is identified with a username. Once the identification is complete, the user needs to be authenticated, which is done with the password. The user’s identity is proven after the user provides the correct password.

After the user is authenticated, the user can access resources on the network on which they are authorized. If the user is not permitted on certain resources, such as files and folders, the user is considered to be unauthorized when he or she attempts to access them. For example, let’s consider an example of two folders, Projects, and Finance, that are shared on the network. The user is part of the Operations team and has access to the Projects folder. After the user logs on to the network, after authorization, the user can access the Projects folder. On the other hand, the user does not have permissions to the Finance folder. When the user attempts to access the Finance folder, the user gets the access denied message.

After the authorization is completed, the user performs certain access on the network resources. If appropriate logging is enabled, the user’s actions are tracked and logged into the logs on the server where the user is connected or performing actions. Alternatively, the logs may be collated in a central environment. The logs are used for auditing purposes.

Cloud vs. On-premises Authentication Requirements

Fundamentally, the process of AAA remains the same in the cloud or on-premise environment. As stated earlier, the user must be identified and authenticated before being authorized to access network resources. Similarly, the cloud also requires user identification and authentication.

In the local environment, you have directory services, such as Active Directory, that handle the identification and authentication part. In the cloud environment, you have Directory-As-A-Service (DaaS) that handles user identification and authentication. However, in most of the cloud environment, you also have Identity and Access Management (IAM) that can handle not only the user and group creation but also the access management for them. For example, Amazon AWS offers AWS IAM, which allows you to create and manage users and groups. Using AWS IAM, you can also enable multi-factor authentication and integrate it with your organization’s directory services.

Similarly, IAM can also be installed and configured in an on-premise environment. However, one has to remember that there are additional overheads when dealing with on-premise infrastructure. You would need to procure servers as well as a license for IAM.

Technologies

In most cases, users are forced to provide a password to authenticate themselves. However, various other methods can be used to add another authentication method along with the password. Let’s look at some of these methods that can be used for authentication.

Time-based One-Time Password (TOTP)

TOTP is a password that has a timestamp and is valid only for a specific duration. For example, a TOTP is valid only for 30 seconds, and after 30 seconds are over, the TOTP expires and cannot be used anymore.

HMAC-based one-time password (HOTP)

HOTP uses hash-based message authentication codes (HMAC) functions to create a one-time password. HOTP uses a secret key and also an incrementing counter. Then, the HMAC creates the hash value, which is then converted into a 6-digit or sometimes 8-digit HOTP value.

Short Message Service (SMS)

SMS is used for sending short messages up to 160 characters. A mobile device can send and receive messages. SMS is the key delivery methods for OTPs or TOTP’s. OTPs and TOTPs are time-bound messages that are delivered via SMS and expire after a certain time period. For example, assume that you are attempting to log on to your bank’s website. You need to authenticate yourself with your username and password. The website will then send an SMS with an OTP to your registered mobile device’s number which provides a second level of authentication which is also referred to as Multi-factor authentication (MFA). The OTP needs to be entered to complete the authentication process. A time limit is associated to the OTP and can vary depending on the application being accessed.

Token key

A token key is also known as a key fob, which is the size of a USB drive. It contains a Liquid Crystal Display (LCD) that displays a number, which is generated by a server. The token key is synchronized with the server, which updates the number every 60-seconds. When you need to authenticate using a token key, you need to provide the user credentials and the number generated. This combination of username, password, and the number is then used to authenticate the user. However, if the number changes while you are entering it for authentication, the authentication will fail. There is no sequence to these numbers that are displayed on the token key. These numbers are dynamically generated by the server and synchronized with the token keys.

Static Codes

An authenticator app generally generates static codes. When you configure an authenticator app, it generates static codes that you have to store somewhere safe. Static codes are required when you lose access to the app, such as a messaging service, you can use the static code to recover your access.

Authentication Applications

Authentication applications are designed to provide application-level authentication. Authentication applications also support digital signatures. Kerberos is one such example. It is the authentication protocol used by Windows servers running Active Directory Domain Services. Kerberos uses a centralized server that authenticates users to servers and vice versa. It uses an Authentication Server (AS) and a Ticket Granting Server (TGS).

Push Notifications

A push notification is an alert sent to an app installed on a mobile device, which can either be a mobile phone or a tab. Notifications were initially used for alerting the user about an Email that has been received on the mobile phone. However, later, push notifications were adopted as a marketing strategy and are now used with almost all apps installed on a mobile device.

For example, let’s consider that you have WhatsApp installed on your mobile. When a message is received, a notification is generated by default. When you install an app on a mobile device, the notifications are automatically enabled for the app. However, in the Settings section, you can navigate to the notifications section and disable notifications for a particular app.

It is important to note that push notifications are app-driven, not the mobile make or the mobile platform. For example, push notifications will work on an app, whether it is installed on an Android phone or an iPhone.

Phone Call

A phone call can also be used for authentication. Some mobile apps, when you are setting them up on the mobile device, may trigger a call on your registered mobile number. After you verify yourself, the app will then start working.

An example can be an app that uses a phone call as an authentication method. For example, let’s consider that you stay in an apartment building. When a visitor comes, the security guard feeds the information in an app, which triggers an automated call to your registered mobile number. You have to press a key to either authorize the visitor to enter or press another key to reject the incoming visitor request.

Exercise 3 - Biometric

Biometric authentication utilizes a physical characteristic of an individual to authenticate him or her. For example, authentication of an individual can be done using a fingerprint or a retina scan. Once the user is authenticated, the user is authorized to access the system or the data within a system. Biometric authentication is widely used at the entry point of a room or even with laptops. Several latest laptop models are equipped with fingerprint authentication. Many mobile manufacturers have integrated the fingerprint and face scanning methods to authenticate a user.

The biometric process has two parts to it. The first part is the enrolment in which a physical characteristic of an individual is registered. The system does not store the physical characteristics but converts it to a numerical value and stores it in the database. The second part is the authentication. When a user uses the physical characteristic for the authentication purpose, it is again converted to the numeric value and compared with the stored value in the database. If both the values match, then the user is authenticated.

In this exercise, you will learn about various biometric authentication methods.

Fingerprint

Fingerprints can be used for identification or even authentication. When you are entering a data center, you would typically see a biometric system installed. In most cases, it is the fingerprint scanner that authenticates a user, and once the user is authenticated, the door to the data center opens. Mobile devices are also equipped with fingerprint scanners. Some mobile devices have a fingerprint scanner on the back, but several new models have a place for the fingerprint scanning on the screen itself. The sensor inside the screen scans the fingerprint and authenticates the user.

Retina

The retina scanner scans the pattern of the blood vessels of the retina. Each individual has a unique blood vessel pattern. After a user’s retina is registered with the retina scanner, the same pattern is detected during the authentication. If there is a variation in the pattern, the user is not authenticated.

Iris

The iris scanner scans the iris pattern of an individual. If the pattern is matched, then the user is granted access.

Facial

A human face has certain features, such as the shape, size, and position of the ear, nose, and eyes. In the facial recognition process, the face is scanned for the facial features and their size and positioning. Most of the recent mobile models (mid-range and high-end) have facial recognition features.

Voice

Each human voice has a certain pattern, pitch, and modulation. In the voice recognition feature, these are compared against the values stored for that individual.

Vein

Vein recognition is also known as vascular biometrics, which is used to scan the veins in fingers, hand, or even in the eyeball. The layout of the vein is stored and then later compared when the user attempts to authenticate himself or herself.

Gait Analysis

Gait analysis is a method of discovering who you are. Each individual has specific body measurement and anatomical factors. Gait Analysis is largely used as a method to detect a specific individual. When an individual walks, there is a specific style each individual has. Using the Gait analysis, a person can be recognized and detected.

False Acceptance and False Rejection Rates

False Acceptance Rate (FAR) is a method that determines the probability of the number of incorrect requests that will be accepted by a biometric system. The formula for calculating FAR is:

FAR = number of incorrect accepted requests/number of authentication attempts

The False Rejection Rate (FRR) is a method that determines the probability of the number of incorrectly rejected authentication attempts by a biometric system. The formula for calculating FRR is:

FRR = number of false rejections / numbers of authentication attempts

Crossover Error Rate(CER)

The CER or Crossover Error Rate occurs when the accept (FAR) and reject rates (FRR) are equal or close to each other. In this scenario, the biometric system is performing optimally. It is always good to have both FRR and FAR close to each other. If they are close to each other, this means that the biometric system is working as it should be.

Exercise 4 - Enable Multifactor Authentication

When a user attempts to log on to a local system or a networked system, the user needs to be first identified and then authenticated. In the authentication process, the user has to provide a password along with the username, which is part of the identification process. Once the user is authenticated, the user can access the system or the resources on which the user has been assigned permissions. With the username and password combination, single-factor authentication occurs.

When a user is asked to enter the user credentials and also use another method to complete the authentication process, then it is called multifactor authentication. For example, on a system, a user provides a username and password and is then prompted to use the smart card. This is an example of multifactor authentication.

In multifactor authentication, you can use different factors, such as something you have or something you know. You can also use attributes, for example, something you exhibit.

To learn more about how to configure multifactor authentication in Windows, please refer to your course material or use your preferred search engine to research this topic in more detail.

Factors

Multifactor authentication mechanism comprises multiple authentication factors, which can include two or more of the following:

  • Something you know (the knowledge factor): This includes the password and also the answers to the secret questions. A user must provide this to be able to authenticate himself or herself.
  • Something you have (the possession factor): This includes something that you have. It could be a one-time password (OTP), a security token, or even an identity card.
  • Something you are (the inherence factor): This includes the biological traits, which are used for authentication. It could include a retina scan, fingerprint scan, iris scan, voice recognition, and face recognition.

You can use a combination of something you know and something you are. This makes authentication stronger than simply using something you know, which is a password typically.

In recent times, location factors are also being used for multifactor authentication. Most smartphones have a GPS feature, which makes this authentication possible. It relates to somewhere you are - your location is used as authentication. For example, a user is allowed to work from home, and his or her home location is considered the authentication factor.

It is also possible to use the time factor as another authentication factor. For example, a user cannot access his or her account outside office hours. This prevents any misuse of the user’s account outside the office hours.

Multifactor Authentication Technologies

Some of the key multifactor authentication technologies are:

  • Security tokens
  • Soft tokens
  • Mobile authentication
  • Biometric authentication methods - retina scans, iris scans, fingerprint scans, finger vein scans, facial recognition, voice recognition, and so on.

Attributes

Other than the factors, certain attributes can be validated. These attributes are:

  • Somewhere you are - It is an address where you can be located. It uses Geolocation technology to identify your location.
  • Something you can do - Something you do is based on your habits, such as your signature that can be validated. It could also be how you type on the keyboard. Windows 10 supports picture-based passwords. You have to add gestures to the picture and identify them when you want to unlock the system.
  • Something you exhibit - It could be a trait that can be scanned and validated, such as a neurological trait, which can be scanned using the MRI methods.
  • Someone you know - It could be someone who can be validated by a chain of trust.
