Exercise 1 - PKI Fundamentals

The Public Key Infrastructure (PKI) is a hierarchy of Certificate Authorities (CA) capable of issuing and signing digital certificates. These certificates are used for validation, data protection, and encryption for networked elements such as web servers, users, and many other resources. The certificates can be issued for various purposes, such as filesystem encryption to Email encryption. Depending on your requirements, you can generate an appropriate type of certificate for use.

In this exercise, you will learn about the PKI concepts.

PKI Concepts

Before starting to work on PKI and certificates, you need to be familiar with various terms, which would help you gain a good understanding of the PKI functionality. Let’s now learn about some of the key terms.

Key Management

It is quite common to use encryption for encrypting data. However, while attempting to safeguard the data, the users forget to protect the keys used to encrypt the decrypted data. The private key, which is the critical component, needs to be managed properly, safeguarded, and protected from falling into a threat actor’s hands. An individual or organization must practice proper key management practices, which include storage, key handling, and its usage.

Certificate Authority (CA)

A CA is responsible for the entire certificate management lifecycle and can issue or revoke certificates. Depending on the type of certificate required, a CA can issue an appropriate type of certificate to users and devices.

The hierarchy begins with a Root CA at the top and with one or more levels of intermediate CA’s below it. In a simple model, an intermediate CA is trusted by the authority granted to it by a Root CA. However, there may be many levels of intermediate CA’s in an actual PKI hierarchy.

For example, a web server operator needing a certificate for SSL or TLS encryption would send a CSR (signing request) to an “intermediate” CA. A Root CA does not generate a certificate for the users and devices. It is responsible for generating the certificates for one or more intermediate CAs.

In some scenarios, you can even create a self-signed certificate, although outside elements will not trust it, and validation errors may be raised. These are mainly used for testing.

Intermediate CA

A large enterprise can create its own Root CA, which can delegate signing authority to intermediate CA servers. A web browser looks at the domain name of a site it visits. It then begins the process of chaining back from one signed certificate to another until it reaches a root certificate signed by one of the certificates in its root store. If it finds a root certificate in this manner, the site’s certificate is accepted.

Registration Authority (RA)

When a client requests a certificate, it is the Registration Authority (RA) to validate the request. If the validation is successful, then the RA confirms to the CA that a certificate can be issued based on the client’s request. It is important to note that the RA never issues a certificate to the client. It is only responsible for validating and sending the request to the CA.

Certificate Revocation List (CRL)

A CRL is a list that contains the names of two different types of certificates:

  • Invalid certificates, which can include the expired certificates
  • Revoked certificates, which have been revoked by a CA

A CRL contains the reason for the revocation of certificates along with the dates when revocation took place. The CA creates a certificate that maintains the CRL. To avoid anyone modifying the CRL, digital signatures are used.

Certificate Attributes

A certificate has several fields with information. These fields are known as the attributes that define the certificate. For example, it has a Subject field, which provides the DName of the client who owns the certificate. The DName field relates to the Directory Name of the object. An attribute can be made of attribute-value pairs, known as Relative Distinguished Names (RDNs). For example:

  • CN: CommonName
  • OU: OrganizationalUnit
  • O: Organization
  • S: State
  • C: Country
  • L: Locality

For example, a certificate can have the following DName:

CN=PLAB Cert, OU=IT, O=Practice Labs, L=London 4, S=London, C=UK

Online Certificate Status Protocol (OCSP)

The OSCP protocol provides the online revocation services, which help a client validate a serial number and provide the certificate's status to the client. When a client sends a request to check the certificate’s status, OSCP’s responder service checks the status and shares it with the client. It could report the certificate status as valid, revoked, or unknown.

Certificate Signing Request (CSR)

When a user requests a certificate from a CA, the request goes in the form of a CSR, which contains the RSA-based public key. The CSR also contains the required information that is needed for the certificate. First, you need to generate private and public keys. When a CSR is sent, the public key is sent in it. The CA takes the public key and embeds it into the certificate that is issued to the user.

Common Name (CN)

A CN represents the Fully Qualified Domain Name (FQDN). When you create a CSR, you need to provide the CN, which should be in the form of www.practicelabs.com or practicelabs.com.

Subject Alternative Name (SAN)

A SAN is a multi-purpose field in a certificate. For different entities, it can serve different purposes. For example, when using an SSL certificate, it can represent different domain names to which the certificate applies. In the given exhibit, notice that SAN is applicable for the SSL certificate used for *.google.com.

For entities, such as a user, a SAN can represent the User Principal Name (UPN). For a computer or machine, it represents the FQDN.

Expiration

Each certificate has an expiration date. When you have a certificate and a lot of other information, you would also see the creation and the expiration date. When a certificate expires, it cannot secure the transactions taking place on the Website to which the certificate was attached. When you load the Website with an expired certificate, the Web browser is likely to display a certificate expiration warning.

Online vs. Offline CA

A CA can be kept as online or offline depending on its role and how frequently it needs to sign a certificate. A root CA should always be kept offline. You would not need a root CA to sign certificates regularly. The term offline means that the CA is not actively generating certificates and is part of the network either in the shutdown state or just not connected.

A root CA would sign the intermediate CA certificates. Other than this, they have little role to play and must be secured, which can be done by keeping them offline. An online intermediate CA can handle CRLs, certificate validation requests, and managing and distributing the certificates.

Stapling

When a user sends a certificate request, the Responder information is also sent along with it. Imagine a scenario in which a CA has to cater to thousands of requests simultaneously. It can increase the load on the CA. Therefore, to reduce the load of the server, the certificate request is sent with the Responder information. This makes it a single request rather than two separate requests. This process is known as stapling.

Pinning

Pinning is the process in which a digital certificate is hard-coded within the application. Consider a scenario in which a mobile connects to different networks as it moves around the city. To ensure that the apps are not compromised, a digital certificate is used. When a server needs to be validated, its public key is validated against the certificate's hashed public key.

Trust Model

In a trust model, several components, such as protocols, services, and devices, work together to achieve a certain level of protection. The trust is easy to establish if it is a single domain. However, there has to be a trust anchor if the trust needs to be between two or more domains.

For example, if you and another user need to communicate with each other using encrypted Emails, you may not trust each other. However, when you see another user’s certificate, which has been created by the same CA as yours, you would automatically develop the trust. This is the case of a single domain where the same CA creates the certificate.

However, there are chances that your CA may have to trust another CA. The trust between two CA is developed using a trust model, which can be of the following types:

Hierarchical: This is the most widely used trust model. There is a root CA, then the intermediate, and then you can also have the leaf CA, which is at the bottom of the hierarchy. The entire hierarchy shares the root CA certificate and the public keys.

Peer-to-peer: In this model, there are CA, but they are not subordinate to each other. Each CA issues certificate to its entities. If a cross-certification between two CA is required, then they need to certify the public key for each other. When they certify the public key for each other, a bi-directional trust is created.

Hybrid: An organization may need to communicate with partners, vendors, or even client organizations. Each of them can have their own CA, which is using the hierarchical trust model. However, to establish a trust between these CA, you can implement the peer-to-peer trust model, where each CA certifies the public key for each other.

Key escrow

For a user, private key management can be a difficult task. To relieve the user from managing and securing the private key, you can use the key escrow process, which uses a third-party to manage the user's private key. In this process, the user’s private key is split into two parts, which are encrypted. Each part of the private key is then stored in a separate location. When the user needs to use the private key, both the parts are retrieved and combined to create a copy of the private key.

In the key escrow process, a copy of the private key can always be retrieved. This process is also helpful when you have lost the private key and need to retrieve it.

Certificate Chaining

In a certificate chain, the certificates trust each other. The issuing of a certificate starts with the root certificate, which is issued by the root CA. The root certificate is then used to sign the intermediate certificate, which signs the certificate to the user or any device. The user or device certificate trusts the intermediate certificate issued by the intermediate CA. The intermediate certificate trusts the root certificate. This creates a chain of trust in which the certificates are chained.

Types of Certificates

Certificates can be broadly categorized into various categories. Some of the key ones are as follows:

Wildcard

In a normal scenario, a certificate is issued to a single entity. Let’s assume a certificate is issued for practicelabs.com. In this scenario, the certificate is valid only for practicelabs.com. In a scenario where you have several entities, such as www.practicelabs.com, web.practicelabs.com, cdn.practicelabs.com and several more such examples. You require a certificate for all these entities.

In such a scenario, you have two choices. The first choice is that you can get an individual certificate for each of these entities. Another option, which is better and more manageable, is to get a wildcard certificate. The certificate would be issued to *.practicelabs.com, which would be valid for entities, such as www.practicelabs.com, web.practicelabs.com, cdn.practicelabs.com and so on.

Subject Alternative Name (SAN)

A Subject Alternative Name (SAN) digital certificate is more commonly known as a UCC certificate. UCC stands for Unified Communications Certificate. A UCC certificate is used with unified communication applications, such as Microsoft Exchange Server.

Code Signing

When developers write the code for an application, they need to protect the code. To be able to do this, the developers can use digital certificates and sign the code. Once the code is signed, it prevents the code from being modified in an unauthorized way.

Self-Signed

Imagine that you have an internal Webserver that you use for testing purposes. You want to integrate a certificate into this Webserver but want to avoid spending money. To meet these requirements, you can create a self-signed certificate. You, as the creator of the certificate, attest it so that the Webserver can use it. A self-signed certificate is an individual certificate and does not have a certificate authority to work with. There are several network devices and applications that use self-signed certificates.

Machine/Computer

Using a certificate authority, you can create a machine or computer certificate, which can be used to verify its identity. With a certificate, a network device can confirm an authentic device and provide services for which it was commissioned. A machine or computer certificate is used to authenticate it over the network.

Email

You can also create an Email digital certificate, which a user can use to sign and encrypt outgoing messages. The same certificate can be used to decrypt the incoming encrypted messages.

User

A user certificate identifies a user. For example, when a user encrypts a hard drive using the Encrypting File System (EFS) and later wants to decrypt it, the certificate is used to authenticate the user.

Root

A root certificate is a certificate that is created by the root CA. It is the first certificate in the trust chain. When a root CA creates a certificate, it self-signs it, which becomes the root certificate.

Domain Validation

Domain validation (DV) certificates validate an organization’s permissions to use a specific domain name. This type of certificate validates the identity of the organization that controls the domain name.

The domain validation certificate displays a padlock before the domain name. You can click on the padlock to view the certificate information.

/

Extended Validation

An Extended Validation (EV) certificate is the enhanced version of the domain validation certificate. The CA has to perform various checks, such as the authenticity of the owner, the organization, physical address, and the organization’s operations. The Website owner must also prove its identity and its ownership of the domain name. Just like the domain validation certificate, EV certificates also add a padlock and company name before the Website name in a Web browser’s address bar.

Certificate Formats

When creating a certificate, you can encode them in different formats. Each encoded format can have a different extension. Here are the certificate formats:

Distinguished encoding rules (DER): A CRT extension denotes a certificate file encoded as a DER binary file or as an ASCII file. If a certificate has the .der extension, then it is in the binary form. A KEY file may also be encoded as a DER binary file or as an ASCII. The DER format contains a key piece of information, which is the binary data. It is mostly used as a single certificate.

Privacy enhanced mail (PEM): This format is used for encrypting Email messages and ensures confidentiality and integrity. A PEM-encoded CRT file is equivalent to a PEM encoded CER file.

Personal information exchange (PFX): It is a file format that is used for authentication of Websites as well as applications. This type of certificate contains both the private and public keys. A password also protects it.

.cer: A CER extension indicates a Microsoft version of a CRT certificate file.

P12: The P12 format is based on the standards defined by RSA Corporation. It also contains both the keys, public and private.

P7B: This format has the extension .p2b or .p7c. This format only contains the public key, but not the private key. It can also contain certificates, a certificate chain, and CRL.

Exercise 2 - Create a Self-signed Certificate

A certificate is a method for securing communication between users and a Web server or between two entities. If a certificate, which can be SSL or TLS certificate, encrypts the communication during transmission, it is hard to decipher the information after capturing it. If a certificate is not used, the information over the Internet travels in a clear-text format and can be intercepted easily. SSL is now obsolete and has been replaced by TLS. You can either use a third-party certificate or create a self-signed certificate.

In this exercise, you will learn to create a self-signed certificate.

Exercise 7 - Implement Key Archival

There can be scenarios in which you need to recover an issued certificate. To be able to do that, you need to configure the CA for certificate recoverability. When a key is lost, you need to have methods in place that can help you recover the lost key. If the key is lost and cannot be recovered, then any encryption done with the key cannot be undone. Therefore, it is important to implement key archival so that the key can be retrieved if it is lost or corrupt.

In this exercise, you will learn how to set up a key archival by first enabling a key recovery agent in certification authority and issue a recovery agent certificate to the CA administrator.
